I used a mix of coding, testing, and review tools throughout the task tracker work. The main help came from the editor-based coding assistant for fast implementation: it helped me edit the frontend and backend in place, add the search/filter API behavior, wire due-date handling into the task modal, and update the card layout quickly. I also used the terminal and pytest workflow to verify behavior directly, which was important because it kept the implementation grounded in real outcomes rather than just code changes. The live API checks were especially useful for confirming that the new query parameters behaved correctly and that invalid filter values returned the expected validation errors.

One moment where AI helped a lot was when I needed to add the search and filter flow across both the backend and the frontend. It was faster to generate the request/response structure, update the rendering logic, and keep the UI aligned with the API than to build everything manually from scratch. Another useful moment was when I was adjusting the task-card styling; AI helped me reorganize the layout so the title and description became the main visual emphasis while the metadata stayed compact on the right.

One moment where AI slowed me down was when it suggested more elaborate UI ideas than the project needed, such as over-designed filter panels or richer reminder logic. Those ideas were plausible, but they added unnecessary complexity for a small in-memory Kanban app. I had to review and narrow them down to keep the implementation simple and aligned with the user’s request.

One place where my review changed the result was the empty-state behavior. The first version of the UI was functional but not very clear when no tasks matched; I corrected it by adding a visible alert message so the user would understand that the filters were working rather than seeing a blank or confusing board. That review step made the feature more usable and more aligned with the intended experience.
